## 1. Executive Summary

The codebase is in good health after the hardening sprint. All baseline checks pass. The primary risks are:

1. **Two P0 documentation errors** that will cause runtime `TypeError` or user confusion immediately.
2. **One P1 behavioral divergence**: async `call_all()` silently skips schema GET validation that the sync client enforces.
3. **One P1 correctness gap**: `is_csv` is not cached, causing repeated `splitlines()` on large response bodies.
4. **One P1 safety gap**: `max_concurrency=0` deadlocks the async event loop silently.
5. **One P1 concurrency hazard**: `extract()` holds a semaphore slot per-target (not per-request), which can starve `get_all_fast()` when both share the same client.
6. **Release is blocked** by an empty `[Unreleased]` CHANGELOG section and undocumented `BlestaEnvConfig` public API.

Overall: **Release-ready after P0/P1 fixes.** No data loss, no billing safety regression, no broken tests. The remaining P2/P3 items are maintainability debt.

## 3. Agent Findings Summary

### Agent A — Sync client
- **Health:** Sound. 100% coverage. No correctness bugs found.
- **Key findings:** SDK_USAGE.md incorrectly says `call()` "defaults to POST" on ambiguous methods (it raises `ValueError`); `allow_http`/`discovery` omitted from constructor docs; `conftest.py` fixture breaks when `BLESTA_API_URL=http://...`.
- **Drift risk:** `DEFAULT_TIMEOUT` and `_IDEMPOTENT_METHODS` constants are duplicated in both clients.

### Agent B — Async client
- **Health:** Functional but has five sync/async behavioral divergences, three of them unintentional.
- **Key findings:** Dead `self._last_request` instance attr; `call_all()` skips schema validation; `count_for()` drops warning log; `call()` handles ambiguity differently; `max_concurrency=0` deadlocks event loop.

### Agent C — Shared modules
- **Health:** Good. All shared helpers are well-tested. Two behavioral divergences confirmed (C-01, C-02).
- **Key findings:** Confirms B-02 (async `call_all` skips schema GET check — behavioral bug); SENSITIVE_KEYS contains four overly-broad keys (`key`, `pass`, `card`, `token`); stuck-page detection doesn't respect `on_error='raise'`; `PaginationError` not under `BlestaError` hierarchy.

### Agent D — Performance
- **Health:** No O(n²) paths in normal use. Two P1 issues found.
- **Key findings:** `is_csv` property not cached — repeated `splitlines()` cost; `extract()` semaphore held per-target starves `get_all_fast` pages; `hash(str(data))` O(page_size) per page; `list.pop(0)` on rolling window should be `deque`; benchmarks not integrated into CI.

### Agent E — Documentation
- **Health:** Largely accurate, but two P0 and five P1/P2 mismatches confirmed.
- **Key findings:** `generate_capabilities_report(format=...)` → runtime `TypeError` (wrong param name in 3 places); `_env_config.py` docstring still has wrong `submit()` call; `retry_mutations` docs imply 5xx retry for mutations (explicitly prevented in code); `BlestaEnvConfig` has no README entry; constructor signatures missing `allow_http` and `discovery`.

### Agent F — Test quality
- **Health:** Behavior-focused, high parity. Seven substantive coverage gaps found.
- **Key findings:** Async injected-discovery path untested; `get_all_fast` per-page error/null branches untested; `AsyncBlestaRequest` ImportError message untested; `_load_bundled_schema` error paths untested; two brittle tests using private internals (`_csv_cache`, `HTTPAdapter._pool_connections`).

### Agent G — Packaging/CI
- **Health:** Mostly sound. Two release blockers identified.
- **Key findings:** CHANGELOG `[Unreleased]` empty; `BlestaEnvConfig` undocumented; publish workflow has no test gate; CI matrix missing Python 3.10/3.11 (declared in classifiers); third-party actions not SHA-pinned.

## 7. Duplication Review

| Question | Answer |
|---|---|
| What duplication remains? | `DEFAULT_TIMEOUT`, `_IDEMPOTENT_METHODS` constants; ~60-line retry loop body; `_validate_segment` static method wrapper; URL construction `f"{self._url}/{model}.json"` |
| What duplication is acceptable? | Retry loop body — differs only in `time.sleep` vs `asyncio.sleep`; extracting this into a shared callback abstraction would obscure intent. Keep separate, document the relationship. |
| What duplication should be extracted? | `DEFAULT_TIMEOUT`, `_IDEMPOTENT_METHODS` → `_constants.py` (trivial, eliminates drift risk). `_validate_segment` wrapper → call `validate_segment()` directly. |
| What should NOT be extracted? | Retry loop logic, transport construction, connection pool config. Different enough in semantics to warrant staying separate. |
| Recommended shared modules | `_constants.py` for timeout/method constants. `_retry.py` only if retry logic actually needs to change in both places simultaneously. |

## 8. Performance Review

| Question | Answer |
|---|---|
| What is already performant enough? | Auth construction, URL building, session reuse, pagination state machine at typical page sizes (20–25 items), schema caching via `lru_cache`. |
| What may become slow at scale? | `is_csv` repeated `splitlines()` (fix: cache); `hash(str(data))` per page at 500+ items/page; `list.pop(0)` on window (fix: deque); full memory materialization of multi-page results in `get_all`, `get_all_fast`, `csv_data`. |
| What needs benchmarks? | `hash(str(data))` vs O(1) fingerprint at 500+/1000+ items/page. Semaphore granularity cost in `extract()`. `get_all_fast` batch size tuning. |
| What should be optimized now? | `is_csv` caching (P1 correctness fix, one-liner). `deque` for pagination window (fold into pagination consistency lane). |
| What should wait? | `hash(str(data))` replacement (benchmark first). Streaming `iter_csv_data()`/`iter_report_series()` (additive API, not a regression). Discovery threading lock (only matters for multi-threaded sharing of custom discovery instances). |
